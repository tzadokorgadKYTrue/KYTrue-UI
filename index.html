Biological identity token

    
The kyt_id — a credential that cannot be stolen, forged, or disputed.

    
Every verified KYTrue session produces a single output: a kyt_id , a unique numerical identity token anchored to a biological event that happened at a specific moment in time. Unlike a password, a document, or a cookie, the kyt_id cannot exist without the living person who generated it.

  


  
  

    

      
Token format

      

        kyt_id_A7Q9K2PN_5F3D
      

    

    

      

        
Segment 1 — RANDOM8

        
8 characters of pure entropy

        
Generated from 40 bits of cryptographic randomness using crypto.randomBytes(5) at the moment of issuance. Encoded in an unambiguous base32 alphabet , no zeros, no ones, no letters that look like numbers. Collision probability: 1 in 1,099,511,627,776 , effectively unique across all verifications, globally, forever.

        
entropy: 40 bits · space: 32⁸ = 1.09 × 10¹² tokens

      

      

        
Segment 2 — CHECKSUM4

        
4 characters bound to the biology

        
Derived deterministically from SHA-256 of four inputs fused together: the haemo_signature_hash (the irreversible fingerprint of this person's blood-flow pattern), the LiveScore, the session ID, and the exact millisecond of issuance. The first 20 bits of the hash become 4 base32 characters. Change any single input , the score, the session, the biometric , and you get a completely different checksum.

        
SHA-256( haemo_hash · score · session_id · issued_at_ms ) → bits[0:20]

      

    

  


  
  

    

      

        
      

      
One token per biological moment

      
The CHECKSUM4 is recalculated fresh on every session. Even if the same person verifies twice, the second kyt_id is completely different , a different session ID and a different timestamp produce a different hash. There is no way to replay, reuse, or recycle a kyt_id.

    

    

      

        
      

      
Issued only above the threshold

      
A kyt_id is minted only when LiveScore ≥ 750 , meaning the verification returned Verified or Trusted. This matters: a high score means the person is genuine. A low score means suspicion or failure. Possession of a kyt_id is proof that a verified human cleared the biological bar at the Verified level or above.

    

    

      

        
      

      
Attachable to any transaction

      
Once issued, the kyt_id travels. It can be embedded in a payment message, a contract signing, an agent delegation, or a CBDC transfer. Every downstream action carries the proof of identity back to its biological source. The chain is irrefutable , the kyt_id in a transaction log is a direct link to the living person who was scanned.

Immune

      
The token is tied to a session ID and a haemo_signature_hash that only exist in KYTrue's audit log. Any re-verification attempt produces a new session , the checksum no longer matches. The stolen string is cryptographically orphaned.

Immune

      
CHECKSUM4 is SHA-256 of inputs that only KYTrue possesses at the moment of scoring: the haemo_signature_hash, the exact score, the session ID, and the millisecond timestamp. Without all four, generating a valid checksum is computationally infeasible.

    


    

      
Transfer a kyt_id to another personGive your verified token to someone else to use in their name

      
Immune

      
The kyt_id is anchored to a specific biological event , a scan of a specific person's haemodynamic pattern at a specific moment. Partners can re-verify the token against KYTrue's audit API at any time. A different person re-verifying will not match the stored haemo_signature_hash.

Immune

      
The Revealense engine extracts a haemodynamic blood-flow signature from sub-dermal vascular patterns. A deepfake can fabricate facial appearance , it cannot fabricate the blood-flow pattern beneath the skin, timed to micro-expressions, timed to a neurological stimulus introduced mid-session. No haemo_signature_hash → no CHECKSUM4 → no kyt_id.

Immune

      
The haemodynamic signature is time-variant , it changes second by second in ways that are algorithmically predictable for the registered person and statistically impossible to replay. A recorded session from five minutes ago will not produce the same signature as a live scan now. Additionally, a fresh session ID and timestamp are generated for every attempt , a replayed video produces a different CHECKSUM4 from the stored one.

    


    

      
Dispute an authorised transactionClaim "that wasn't me" after a verified action has been executed

      
Immune

      
The kyt_id embedded in the transaction links back to an immutable LiveAudit entry containing: the haemo_signature_hash, the LiveScore, the session recording metadata, the exact timestamp, and the CHECKSUM4 derivation inputs. The audit trail is regulator-ready and legal-hold compliant. Disputing the transaction requires disproving a SHA-256 computation , which is not possible.

Token format

        
kyt_id_[RANDOM8]_[CHECKSUM4]

      

      

        

          kyt_id_A7Q9K2PN_5F3D
        

        

          RANDOM8 — cryptographic entropy
          CHECKSUM4 — biological binding
        


        

          
// RANDOM8: 40 bits of true randomness
// Base32 alphabet — no 0/O, no 1/I, no L, no 8/B
const raw = crypto.randomBytes(5);   // 40 bits
const RANDOM8 = toBase32(raw, 8);     // 8 chars

// CHECKSUM4: SHA-256 of four inputs
const payload = [
  haemo_signature_hash,   // from Revealense
  String(livescore),      // 0–1000
  session_id,             // this session only
  String(issued_at_ms),   // millisecond timestamp
].join('|');

const sha = createHash('sha256')
            .update(payload).digest();
const CHECKSUM4 = toBase32(sha, 4); // first 20 bits

return `kyt_id_${RANDOM8}_${CHECKSUM4}`;

        


        

          Why four inputs in the checksum?
          
    
            
  • haemo_signature_hash — ties the token to this person's blood-flow pattern. An impostor's scan produces a different hash.
    • 
            
  • livescore — ties the token to the score that cleared the threshold. A tampered score breaks the checksum.
    • 
            
  • session_id — ties the token to this specific session. Replaying the session produces a new session ID; the checksum no longer matches.
    • 
            
  • issued_at_ms — ties the token to the exact millisecond of issuance. Even two scans of the same person one second apart produce different checksums.
    • 
          

          
Alphabet: Base32 with the unambiguous set ABCDEFGHJKMNPQRSTUVWXYZ23456789 , deliberately excludes zero, one, letter-O, letter-I, letter-L, and eight, which are visually confusable in monospace fonts and printed documents.

What to do with a kyt_id once you have it

    

      

        Attach to transactions
        
Embed the kyt_id in every payment message, contract, or authorisation event that the verified person initiates. Include it as a header, a metadata field, or an ISO 20022 enrichment. Any downstream system , clearing network, insurer, regulator , can trace the action back to the biological source.

        
Example: ISO 20022 payment → append kyt_id_A7Q9K2PN_5F3D to CdtTrfTxInf/Dbtr/Id/PrvtId. The clearing leg is now non-disputable.

      

      

        Re-verify with step-up auth
        
For high-value actions (large transfers, contract signings, agent delegation), call POST /v1/identities/:kyt_id/verify with the original token and the client_reference. KYTrue runs a fresh live scan and checks biological continuity against the stored haemo_signature_hash. It also validates that the client_reference matches the one bound to this kyt_id at issuance — a biological match alone is not sufficient if the identity anchor belongs to a different record. If either check fails, the re-verification is rejected.

        
Fields returned: biological_continuity (0.0–1.0, threshold 0.97) and client_reference_match (bool). Both must be true for the loop to close.

      

      

        Anchor AI agents
        
Call POST /v1/agents/anchor with the kyt_id as principal_id plus the agent's scope (max amount, merchant categories, expiry). Every action the agent takes in the agentic payment stack carries the anchor token , cryptographic proof that a verified living human authorised the delegation at a specific moment in time.

        
Revocation: if re-verification of the principal fails at any subsequent check, the entire anchor chain is invalidated retroactively across all protocols , MCP, A2A, AP2, ACP.

      

    

  


  
  

    
When kyt_id is null — what it means and what to do

    

      

        score below 750 — Borderline / Suspicious / Reject
        
Strong incongruence detected between physiological and behavioural channels. High-confidence deception or stress markers. kyt_id is not issued. Do not auto-approve. Route to SIU or fraud team for manual review. The LiveAudit entry is available for investigators , it contains sincerity sub-scores and stress marker counts.

      

      

        score 0–399 — Reject
        
Liveness check failed, deepfake injection detected, or haemodynamic response absent. This person is not a living, present, sincere subject at this moment. kyt_id is not issued and will never be issued for this session. Block the transaction. Log the incident. The audit entry is regulator-ready for SAR filing. The session_id can be used to pull the full Revealense analysis via the audit endpoint for legal proceedings.

      

    

    

      The principle: A kyt_id is not a record of an attempt , it is certification of success. The score encodes the quality of the biological signal: high score = strong, clear, unambiguous signal from a living person. Low score = weak, inconsistent, or absent signal. Identity certification belongs only in the first category.

At the moment of a high-value action, a fresh scan is taken. The new haemodynamic signature is compared against the stored haemo_hash. biological_continuity ≥ 0.97 confirms identity closure.

Loop closed or rejected

        
biological_continuity score returned. ≥ 0.97 closes the loop. Below threshold, impostor detected.

biological_continuity ≥ 0.97

          

        

        
The fresh biometric scan matches the haemo_signature_hash stored at enrolment, and the presented client_reference matches the one bound at issuance. The same living person, in the same account context, is confirmed present. The action is cleared. A new audit entry links both sessions: the kyt_id, client_reference, and re-verification timestamp are permanently bound in the audit trail.

How it works under the hood

        
Biological continuity scoring

        
When a re-verification is triggered, Revealense runs a fresh haemodynamic capture, the same 15-second scan used at enrolment. The resulting haemo_signature_hash is then passed to KYTrue's Continuity Engine, which computes a multi-dimensional similarity score against the original stored hash across three axes:

        

        
1. Pattern similarity: the sub-dermal vascular structure is unique per individual. The engine compares the topological fingerprint of blood-flow distribution.

2. Dynamic coherence: haemodynamic patterns are time-variant but person-consistent. The velocity and phase of blood-flow pulses are compared to the enrolled baseline.

3. Stress-response consistency: a neurological stimulus is introduced mid-scan. The autonomic reflex profile of the live subject is compared to the enrolled subject's stored reflex curve. A different person cannot fake this response.

Insurance

        
Claim identity confirmation

        
At claim submission, the policyholder's kyt_id from onboarding is re-verified. If the person submitting the claim matches the enrolled policyholder biologically, the claim proceeds. Mismatches flag potential fraud before a single penny is disbursed.

        
POST /v1/identities/:kyt_id/verify · context: claim_submission

Wallet action authorisation

POST /v1/identities/:kyt_id/verify · context: onchain_tx

Signature verification

              
Every event is signed with HMAC-SHA256 using your webhook secret. Always verify before processing.

              
NODE.JS
const sig = req.headers['kyt-signature'];
const [ts, v1] = sig.split(',');
const payload = ts + '.' + req.body;
const expected = crypto
  .createHmac('sha256', secret)
  .update(payload).digest('hex');
if (v1 !== expected) throw new Error('invalid');
